Dezentrale Versionsverwaltung

mit Git

Warum Git dezentral ist — und warum wir trotzdem einen Server brauchen

Melina arbeitet lokal

Melinas erste Commits

Melina hat zwei Commits erstellt. Alles passiert lokal — kein Server nötig.

Niklas klont Melinas Repository

Niklas erhält eine vollständige Kopie inklusive der gesamten Historie. origin zeigt auf Melinas Repo.

Niklas erstellt einen Commit und pusht

Niklas erstellt C3 und pusht zu Melinas Repository (origin). Beide haben jetzt den gleichen Stand.

Melina committet, Niklas pullt

Melina erstellt C4. Niklas holt sich die Änderung mit git pull. Beide sind wieder synchron.

Gregor kommt dazu

Gregor klont von Niklas und fügt Melina als zweiten Remote hinzu:
git remote add melina <url>

Gregor pullt von Melina, pusht zu Niklas

Gregor holt C5 von Melina und gibt ihn an Niklas weiter. Git ist ein Peer-to-Peer-System!

Warum funktioniert das so nicht?

  • Jeder müsste seinen Rechner als Server bereitstellen (Firewall, dynamische IPs)
  • Wer hat die „Wahrheit“? Welches Repository ist der aktuelle Stand?
  • Was passiert, wenn jemand offline geht oder den Laptop zuklappt?
  • Bei mehr als 3 Personen wird es unübersichtlich — wer hat welche Commits?

Die Lösung: Zentraler Git-Server

Alle arbeiten gegen einen zentralen Server. origin zeigt immer dorthin.
Dezentral in der Technik, zentral in der Organisation.
